Safety & constraints (must follow)

  • Bind only to 127.0.0.1 (never 0.0.0.0).
  • Use the secure modules by default: *_secure.py, secure_network_tools.py.
  • Attack endpoints remain enabled but are simulation only.
  • Only test on networks you own or have written permission to test.
  • Before any code edits, run python3 test_security.py.
  • Review cyber_matrix.db and cyber_matrix.log for audit.

Environment configuration (.env)

Create /workspace/.env with secure defaults.

cat > /workspace/.env <<'EOF'
CYBER_MATRIX_HOST=127.0.0.1
CYBER_MATRIX_PORT=5000
CYBER_MATRIX_API_KEY=change-me-32+chars
LAB_MODE=false
CYBER_MATRIX_DB_PATH=/workspace/cyber_matrix.db
CYBER_MATRIX_LOG_PATH=/workspace/cyber_matrix.log
EOF
set -a && . /workspace/.env && set +a
Don’t have a key yet? Generate and activate via API 
API_KEY=$(curl -s -X POST http://127.0.0.1:5000/api/auth/generate-key | jq -r .api_key)
curl -s -X POST http://127.0.0.1:5000/api/auth/activate \
  -H "Content-Type: application/json" \
  -d "{\"api_key\":\"$API_KEY\"}"
export CYBER_MATRIX_API_KEY="$API_KEY"
curl -s http://127.0.0.1:5000/api/auth/verify -H "X-API-Key: $CYBER_MATRIX_API_KEY"
Note: The server stores a hash in-memory; you’ll pass the raw key via X-API-Key.

    "http": {
      "command": "server-http",

    }
  }

MCP Task Recipes for kam.krk_v2.o

Recipe A — Verify API access via HTTP server

Use the HTTP server to talk to the running Flask app with your API key.

# HTTP GET (send header X-API-Key)
# Method: GET
# URL: http://127.0.0.1:5000/api/auth/verify
# Headers: { "X-API-Key": "$CYBER_MATRIX_API_KEY" }
Tip
If verification fails, (1) ensure the backend is running, (2) set CYBER_MATRIX_API_KEY using the steps in the Environment section.

Recipe B — Edit code with Filesystem, commit with Git

  1. Open a file with FS (e.g., app.py lines 205–212 for sanitize_input).
  2. Edit carefully; keep secure behavior and validations.
  3. Use Git: status → diff → add → commit (local only).
# Git (via MCP git server)
# status
# diff
# add paths: app.py (or changed files)
# commit message: "Improve input sanitization and validation behavior"
Tip
Do not push by default. Keep changes on the current branch until reviewed. Set MCP_GIT_ALLOW_PUSH=true only in controlled environments.

Recipe C — Start a network scan and poll progress (HTTP)

# Start scan
# Method: POST
# URL: http://127.0.0.1:5000/api/network/scan
# Headers: { "X-API-Key": "$CYBER_MATRIX_API_KEY", "Content-Type": "application/json" }
# Body: { "ip_range": "192.168.1.0/24" }

# Poll progress
# Method: GET
# URL: http://127.0.0.1:5000/api/scan/progress
# Headers: { "X-API-Key": "$CYBER_MATRIX_API_KEY" }
Tip
If nmap is not present, results still work with ARP and basic checks. Install sudo apt-get install -y nmap for richer output.

Recipe D — Port scan and basic vulnerability snapshot (HTTP)

# Port scan
# POST http://127.0.0.1:5000/api/port/scan
# Headers: { "X-API-Key": "$CYBER_MATRIX_API_KEY", "Content-Type": "application/json" }

# Body: { "target_ip": "192.168.1.10", "intensity": "medium" }
Tip
Attack routes are simulations; use them for educational flows without real exploitation.

Recipe E — Fix a failing security test

  1. Reproduce by running python3 test_security.py in a terminal.
  2. Use FS to open the relevant function (e.g., sanitize_input in app.py).
  3. Adjust logic minimally; keep regex removing dangerous characters and length limits.
  4. Commit via Git with a clear message focused on the why.
  5. Verify by calling key endpoints with HTTP to ensure no regressions (e.g., /api/port/scan rejects bad input).
Tip
If a test expects specific sanitization, mirror that behavior while preserving security invariants (no shell=True, input length caps, strict IP/port validation).

Recipe F — Capture artifacts for review (FS + HTTP)

  1. After scans, fetch charts/metrics with HTTP for quick summaries.
  2. Use FS to copy cyber_matrix.log and an SQLite backup of cyber_matrix.db into an artifacts/ folder.
  3. Commit artifacts with Git if desired (local audit trail).
# Example artifact steps (run in terminal)
mkdir -p /workspace/artifacts 2>/dev/null || true
cp /workspace/cyber_matrix.log \
  "/workspace/artifacts/cyber_matrix-$(date +%F-%H%M%S).log" 2>/dev/null || true
sqlite3 /workspace/cyber_matrix.db \
  ".backup '/workspace/artifacts/cyber_matrix-$(date +%F-%H%M%S).db'" 2>/dev/null || true

Troubleshooting (no‑fail tips)

Virtualenv errors

  • ensurepip not available: sudo apt-get install -y python3-venv
  • Use without venv: pip3 install --user -r requirements.txt
  • PATH warnings: add export PATH="$HOME/.local/bin:$PATH"

Network tools missing

  • ip not found: sudo apt-get install -y iproute2 (app falls back to psutil).
  • nmap not installed: scans work, but results improve with sudo apt-get install -y nmap.

API key issues

  • Regenerate with /api/auth/generate-key then /api/auth/activate.
  • Always send X-API-Key on protected routes.

DB/Log visibility

  • DB created on first write; logs rotate when DEBUG=False.
  • Keep server single-instance to avoid DB locks.
